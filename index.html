Project 1: Name Tag

This project was fairly simple! I started by downloading the proper 3MF file for the name tag and opening it in Prusa Slicer, which I already had downloaded from my Ancient Technology and Modern Technology class with Professor Ratzlaff last semester. After that, I was able to edit the font and change the name tag so that it actually has my name on it. My print ended up using 4.03 grams of filament and taking a little over 20 minutes, and I printed it with a brown base and switched it to a yellow filament during the filament swap.

As for the actual printing process itself, there was really only a minor hiccup in that the printer was unresponsive at first, but by simply power cycling the machine, I was able to get it up and running properly again in no time. The filament swap was also relatively easy, and I didn’t run into any issues regarding improper feeding or anything. I am really pleased with how it came out!

Project 1: Torture Test

To start this part of the project, I looked around on Thingiverse and Printables to find something that seemed like a difficult print (based off of my rather rudimentary knowledge of 3D printing). After looking through several different models, I ended up finding one that seemed nearly impossible. It had large, unsupported overhangs and included several upside down cones with tiny points of contact to the build plate. In fact, I was even recommended to add a two-layer raft so that there was even a slim chance of it printing somewhat correctly. (Which I did add, not that it ended up helping that much.) After downloading the STL and uploading it to Prusa Slicer, I had to scale it down so it would fit on the build plate. I found making that sort of small edit with Prusa to be pretty simple! I can definitely see myself finding some cool models to print in the future (I’m already planning on printing a deck box for Magic the Gathering soon) and those sorts of simple, in-slicer edits will most definitely be helpful.

When it comes to the physical print, things most certainly went wrong. Just in watching the first few layers, I could immediately see that the sections based off of small points of contact on the plate were already not sticking well to the rafts. I was honestly surprised when I saw well the bridging sections came out, all things considered. Granted, there was still not amazing bridging, as the underside of those sections were a little bit rough and stringy. Other than that, I mostly ended up with a mess of filament spaghetti, which I think is due to the printer trying to print on top of air because the layers that those sections were supposed to print over did not adhere to the bed. Apparently, the print even had to be stopped part way through because it was beginning to form a gloopy slug. Overall, this project was very helpful for me to get an idea of the limits and capabilities of this kind of 3D printing, and I think I will have a better understanding of what these machines are capable of, not to mention that I should definitely use supports when the slicer advises me to!

Project 2: Tinkercad Owlbear

This part of project 2 went really smooth! I decided to print an owlbear miniature for a D&D encounter I plan on running soon. I used Thangs to find the STLs that I wanted to use, which consisted of an owlbear and the rock that it is standing on (separate files but made by the same modeler), a cave-terrain base for the mini, a nest, and an egg model that I copied three times. (The screenshots I’ve included have the eggs and nest hidden by the legs, sorry!)

At a certain point in the process, I would honestly say that this project perfectly bridged the gap between a project for class and a passion project! I used meshlab to simplify the mesh of the owlbear, but otherwise I was able to import all the other models into tinkercad without any issues. Using tinkercad was really easy. I’ve used the program several times before, so it was just a matter of positioning everything how I wanted it, grouping the objects together, and exporting the file as a 3MF.

In Prusa, I had to add in supports for the model. This was pretty easy, as I am already somewhat familiar with the process. However, I had to go back a few times in order to fix the tinkercad model so that all parts were actually touching each other, which got a little tedious by the third attempt.

The printed model had a lot of supports to remove! Luckily, I had used organic supports so they were relatively easy to remove, and there were little to no artifacts from the supports that remained.

This print came out *really* well! I am honestly a little shocked with just how much detail was conferred, despite the mesh for the owlbear having been simplified. I think the only tiny nitpicks I have with the finished result is that the foot of the owlbear is technically not quite touching the rock that it's supposed to be posed on, but it really isn't noticeable and still is extremely stable. Additionally, the base was a little too thick in my opinion, so if I were to remodel it I would probably try to cut off part of the bottom to thin it out. Also, the owlbear itself was roughly half an inch taller than I expected, so I would probably also scale it down slightly in the future, but overall those are nonissues in the long run. The only other thing that I noticed was that there was a teeny tiny little bit of stringing near the nest area, so I lightly used the heat gun to clean that up.

Project 2: Fusion Hill Pit Trap

This part of project 2 was a little trickier. After much deliberation about what exactly I wanted to do, I decided to make a piece of terrain for the aforementioned owlbear encounter for D&D. My plan was to find a cliff or hill model and remove a section to act as an interactive pit trap.

I was immediately faced with my lack of fusion capabilities. While much of the new program was generally intuitive, the only semi-rudimentary understanding I had certainly made performing this operation more complex. After looking through several menus, I eventually found the split body command. After a few failed attempts at using it and a youtube tutorial, I finally figured it out!

The next thing I wanted to do was to try to add an engraved 1 inch grid pattern into the surface of the hill at a uniform depth of about 0.05 inches. This… did not go so well, when I realized that the emboss tool wouldn’t work for this application, I sketched out a grid and tried to extrude the walls of the grid to the surface of the hill and then offset it down to cut in the pattern.

I can't even count how many attempts, errors, and crashes I got when trying to do this part. Eventually, I came to the conclusion that Fusion simply hates the idea of trying to split and reconstruct so many polygons, and thus refuses to do it. Having given up on the grid, I just made sure to add tolerance for the pit section to slide in and out.

I then exported it to Prusa, where I realized it was too big for the build plate, resized it and the hole back in Fusion (I needed the pit section to remain 2 inches by 2 inches), and finally got it sliced properly and got it printed!

This part of the project led to a lot of really good learning for me! While my many hours of continuous failure were certainly frustrating, I think it was very helpful in the long run. I certainly can say that I feel much more confident with using Fusion, and I can really see just how much more powerful of a tool it is than tinkercad.

This part of the project came out really well though! I’m a huge fan of the filament I ended up using; I think the blends of yellows and greens are really cool looking. I’m also really happy with the buttery smooth tolerance between the two parts, and I think that the cragginess of the tallest side is perfect. If I had to do this project again, I would definitely make sure to sketch out the build plate first so that I can make sure to be working within the bounds of the print bed the whole time, and I would probably try to use a less complex mesh so that I can actually do my uniform-depth-engraved grid idea, especially because I definitely didn't actually need as much detail as I used. Additionally, I would maybe try a slightly tighter tolerance to see if I can’t make the cut away section blend in a little better. Overall though, I would say that this part of the project was pretty much a complete success (aside from my inability to add a grid like I had originally planned.

Project 3: Line-Following Robot and Enclosures

The instructions seem pretty straightforward! I would expect the challenges to be the actual process of soldering. While I have soldered before, I have had issues with burning/stripping the board, soldering poor connections, etc. in the past. Something that seems a little unclear from the instructions how the yellow gaskets and wheel pieces attach. I think it would be nice if the instructions used more descriptive language about how to actually install the pieces. For example, if it said “press-fit the…” or “screw on the…” However, I do really appreciate how the directions instruct us to enjoy the effect at the end.

When we soldered, Jake and I took turns soldering the components of our first car, which we dubbed “Bafilda” (left). For our second car, “Bathilda” (right) Jake soldered most of the components on while I troubleshooted Bafilda. After checking every component twice, the issue turned out to be poor batteries, which was frustrating to say the least. The dimensions of boards of the cars were as follows: Thickness - 1.44mm, Inner width - 57.69mm, Outer front width - 72.42mm, Outer back width - 69.52mm, Back indent to front - 100.27mm, and Back indent depth - 4.1mm. We also measured the motors: Depth - 14.87mm, Width - 19.8mm, and Height - 28.25mm.

Next, we met up over February break to work on the external chassis for Bafilda and Bathilda. Unfortunately, however, after a small drop during a test of Bafilda, one of the LEDs detached slightly from the board, stripping off part of one of the traces, so we had to move forward with only Bathilda. We ended up coming up with the idea of a horse mounted to Bathilda via slots on the side of the legs.

Before we began designing that, however, we traced over one of the images of the circuit board from the instructions in Fusion and scaled to match our measurements. There was definitely some trial and error to getting dimensions that matched up, but we eventually got there by scaling the dimensions relative to one of the measurements.

For our first iteration, we uploaded a horse model we found on Thingiverse into Fusion. Upon doing so, we immediately noticed that the horse would have to be scaled in order to fit onto the chassis like we had envisioned. This resulted in us stretching the horse across one axis, making a hilariously wide horse that we ended up dubbing “Hort”. Using the sketch of the circuit board, we were able to cut away a slot in the legs for Hort to attach to the board with.

In regards to actually printing Hort, things went extremely smoothly! It was a much longer and heavier print than I had anticipated, and most of the time seemed to stem from the support material. Overall though, this made sense and resulted in a really nice print.

However, when it comes to mounting Hort onto the board, we realized that I had made a rather large lapse in judgement. I had forgotten that the circuit board had things attached to it, and thus Hort would not be able to be slid on like I had thought. We ended up cutting two of the legs off with a bandsaw in order to mount Hort onto Bathilda, which was not ideal because it was quite finicky to wedge in the legs in a way that remained stable, especially seeing how the legs on that side of Hort’s body were slightly shorter due to the material removed by the bandsaw. Additionally, because of the weight, Bathilda struggled a little bit due to the added resistance.

With this in mind, we moved on to a second iteration of Hort, this time using Prusa’s cut feature and adding connector pins and holes. Doing so was surprisingly easy!

This version of Hort seemed really promising. By using two printers, it was able to be printed at a quarter of the time. We also decided to only use 5% infill to reduce the weight of the print.

Unfortunately, this iteration of Hort also had its fair share of issues. We had neglected to paint on support blockers in the holes for the clips, so most of the holes were full of mostly irremovable support material, so we had to snip off most of the clips. As for the holes that we were able to unblock, the clips snapped off when we tried to disconnect the two sides, leaving us with just two halves of a Hort. While we could have simply glued the two halves together, we really wanted to have a removable design.

To remedy this, we ended up making a version using the plug/dowel style of connector in Prusa. We measured the diameter and depth of a magnet, and sized the holes appropriately. Then, we split the two halves of Hort onto two buildplates within the same instance of Prusa (a first for me). This print went even faster than the clip version of Hort, because both sides were able to be laid down on the cut face, reducing the amount of supports and time needed.

Finally, we ended up with a perfect version! The magnets press fit perfectly into place, and stayed in even when connecting and disconnecting the two sides. Due to the orientation of the print, the sides all came out especially smooth and good-looking. Additionally, the reduced weight gave Bathilda an easier time driving. Hort successfully allows access to all 5 key features: the wheels can move, the sensors can see the ground, the LEDs are all visible, Hort can be removed easily to change the batteries, and the power button is fully pressable.

Next, we moved on to the laser cut version. We started by using the split body feature in Fusion to split the Hort model into several parts, the tail, the back legs/hips, the front legs/hips, and the head/neck. Then we created sketches of the both leg sections by turning the other parts invisible and making projections of each set of legs/hips to capture the outline of them from a front facing view. Then, we extruded those sketches by 3mm to make models of the leg panels. Next, we reassembled the split sections of the original model, and re-split the model to just separate off the legs.

After that, we created a sketch projection of the side profile of Hort, and then extruded it by 3mm to get the body piece. Following this, we were able to make all but the extruded projections of Hort invisible, leaving us just with pieces that could be used as a laser cuttable outline. However, we needed a (non-permanent) way for the pieces to actually fit together. So, we cut slots 3mm wide into the pieces that went halfway deep on each of the intersections of the pieces.

Once that was done, it was simply a matter of exporting the outlines of the pieces to Lightburn and making sure everything was set to cut. We decided to use acrylic as the medium, because the idea of such a simple, 2D-looking design that was already going to be difficult to see also being clear (and thus even less visible) really tickled me.

The laser cut version of Hort came out really well! Laser Hort successfully allows access to all 5 key features: the wheels can move, the sensors can see the ground, the LEDs are all visible, it can be removed easily to change the batteries, and the power button is fully pressable. I am super proud of how all of this project turned out. There was a lot of good problem solving and learning that came out of it, which was frustrating at times, but overall really satisfying. I really feel like this project helped me start to feel more like an engineer, and I was able to start anticipating some problems before they arose.

Project 4: e-NABLE Phoenix Hand V2

Personally, I found the e-NABLE hub to be a more useful site to find resources to build the hand. The guides and files were easier and more directly able to be viewed and downloaded, and overall the site seemed more geared specifically towards the hand we were making for this project. However, I like how the NIH website seems to offer more options for other models, and also can show remixes with design tweaks. I think that both sites could better organize how they are laid out, so that there is a clearer step-by-step process shown. For example, I think that if the links on the e-NABLE site were reordered, it could be less contrived to find the correct resources because for me, it was somewhat difficult to navigate to the resources I wanted. Additionally, it might be easier to find if they were somewhat larger or if the videos were embedded on the page, rather than being external links. Personally, I found the videos from the e-NABLE hub to be a little bit more helpful than the instruction manual. I think this is mainly because it was a little confusing in their instructions exactly how many of each part was needed. To remedy this, all that would be needed is an extra page in the instructions saying how many of each part was required. Overall though, the assembly process was pretty straightforward and intuitive for me, so I don’t think anything really needs to change in that department.

Printing the hand went pretty smoothly! My partner and I were able to divvy up what we printed to avoid too much redundant printing. I had a minor bit of trouble with one of my prints due to some under extrusion on one printer, but after switching printers we were able to resolve the issue and continue the project. The other issue that I had was in laying out my buildplates. At first I was really set on trying to only use two buildplates; however, I eventually came to the conclusion that there was no way everything would fit on so little space, especially with extra space being taken up by rafts and supports. This was a really good lesson about when to stop being stubborn and be more realistic.

Assembly was also really easy to follow. The only minor trouble with assembly was we made a minor mistake when stringing the fingers, in that we forgot to thread it through a set of holes, but that was easily remedied. I also messed up a little bit in a different way, as when I was removing the raft/supports from one of the parts I succeeded a little too well and ended up gouging my hand a bit with the pair of snippers I was using. This was a good learning moment because it was a great reminder to take things a little slower and steadier, and overall be more careful.

For our modification, we wanted to add some kind of phone mount to the gauntlet, because we came to the realization that the grip strength on the hand was good, but not fantastic, so holding a phone for hours out of a day seemed a little unfeasible. So, by adding an adjustable phone mount, we figured that we would be able to make a more comfortable way to hold the phone. In our original sketch we thought about a more compact design, but figured that it would be too difficult to operate with only one fully functional hand, as well as that it would be more restricted to a certain phone, due to the specific measurements. So, after some browsing on Thingiverse, we found a model that could be folded compactly, while being overall more adjustable and user-friendly, as well as easier to remove and maneuver for purposes such as charging, making phone calls, etc.

Overall, this project was a really interesting challenge. It was certainly one of the projects that we did, so it was rewarding to take on the rapid prototyping mindset in this way. This project also felt different because of what we were making. Instead of just creating something to learn, be creative, and fulfil the project guidelines, I found myself thinking about the actual usefulness of the product. I found myself wondering what it would be like to use a prosthetic like the one we made if I were missing a hand or fingers. This really made me think about the elements of empathetic design that is discussed in Sarah Hendren’s What Can a Body Do? How We Meet the Built World. Particularly, the question of who the world is built for (Hendren 28) is really striking to me. Only by thinking in this way was I able to start to see the issues with the hand for that needed modification. I think that this outlook is a really powerful tool for design equity, and I hope to take it forward with me into the next project.

Final Project: Door Accessibility

      Boaz, D., 2019. Design, Doorknobs, and the Great Enrichment, Cato Institute. United States of America. COI: 20.500.12592/3jrcgd.
          As Americans get older and wealthier, interest in "aging-in-place design" is growing. This approach focuses on making homes accessible for people of all abilities, going beyond basic requirements like the ADA. It reflects broader trends of larger, more luxurious homes, driven by increased wealth. While inclusive design improves accessibility, it comes with trade-offs, as seen in examples like replacing knobs with levers for easier use or changes made for pets, showing that no design suits everyone’s needs.
      Wenning, S. (2011). Modifications of Homes for the Comfort of Arthritis Sufferers. : Oregon State University. https://ir.library.oregonstate.edu/concern/honors_college_theses/1c18dh70v?locale=en.
          This study examined the use, awareness, and helpfulness of features that could improve the quality of life for people with arthritis, including features such as lever door handles. However, less than half were aware of such features, and only a small minority had them in their homes. The study suggests that increasing awareness of these features is key to maximizing their potential in improving arthritis treatment and quality of life.

Interviewee 1: Brandeis student from New Jersey who has injured their wrist in the past. Their injury came from past rock climbing and volleyball incidents that have resulted in lasting pain and mobility problems. Due to their wrist injury and residual effects (such as pain), they struggle to open door knobs with their injured hand. The solutions they had found in their daily life have been using their other (non-dominant) hand, and preferring doors with levers or push operation.
Interviewee 2: Brandeis rugby player who injured their wrist over time while playing. They have not been to a doctor, but notice their injury getting worse when falling/tackling and catching. They have door knobs in their dorm, and will often use their non-dominant hand (not injured) to open doors to avoid pain.

Interviewee 3: Brandeis student from Massachusetts with a history of shoulder injuries from various athletic pursuits. (Dodgeball, volleyball, quadball) Has had 2 surgeries to prevent future dislocations of the shoulder. They don’t find door knobs too inconvenient, but agreed that push doors and handles were definitely preferable options.

To start out our brainstorming process, we began by discussing our interviews. Overall, we thought our interviews went really well. All of us felt that we were able to get the information we needed while having enjoyable and natural conversations that seemed to make our interviewees feel comfortable. With this in mind, we moved on to the thematic analysis stage of the brainstorming process. To do this, we took a short period of time to write down aspects, words, notes, etc. related to the interviews that particularly stood out to us. From this, we synthesized the general themes of locations, movement restrictions, required materials, extensions, features, installation, and scope, which we took into consideration going forward into the insight statement phase.

After spending time discussing the themes we decided on, we came up with a list of seven insight statements that addressed our themes. We quickly followed that step by drafting several “how might we” statements in order to come up with areas of design to focus our brainstorming for. To accomplish each of these steps, we set a short timer and separately wrote our ideas on sticky notes until the timer was up, after which we came back together to discuss our insights. I found these phases of the brainstorming process to be really beneficial. Not only were we able to explore many avenues of thinking, but were able to simultaneously widen and narrow our focus in a really productive and interesting way.

Next, we used our “how might we” statements to create categories for our brainstorming. Similarly to the previous phases, for each category, we set timers and individually wrote our ideas on sticky notes. We followed that by moving to a white board and putting up all of our ideas so we could all see and discuss them.

We then had a lively and respectful conversation where we discussed our own and each other's ideas, and grouped them together or eliminated them with consensus. After this process, we were left with a much more focused area of work. We ended up with the synthesized idea of creating a 3-D printed attachment that would clamp onto door knobs in order to convert them into lever door handles for both convenience and to facilitate better door usage for people with arm injuries.

The design and prototyping process for creating accessible door accessories has primarily been testing existing models found online and interviewing some users in one of our target demographics to understand their likes and dislikes of the existing solutions. In our interviews, we received many pieces of feedback, positive and negative, all helping us refine our idea. One of our primary takeaways from these interviews was that users did not want to have something that needs to be put on and taken off every time they want to open the door, however, having something portable would be nice in situations where the door knobs are sticky and the doors are heavy.

Another takeaway from this process was that being able to open a door with your elbow is very useful, not just for those with mobility issues, but also if your hands are full, being able to use your elbow to open a door is very useful. From this piece of feedback, we decided to make sure the handle would stick out far enough to ensure a person would be able to put sufficient pulling force on the door to turn the handle and open the door. Not all of the feedback was positive though, in one of our interviews, we received the feedback that the handle did not feel secure on the door. This one is particularly important, as the handle needs to be able to secure to all kinds of door knobs while still being easily removable when necessary.

Another comment about the stability of the handle was less about the securing method to the door, and more about feeling secure when grabbing the handle to open the door. It was suggested that we add a curve to the end of the handle, making an almost L shape to ensure a hand cannot slide out of the handle while twisting it. This L shape would also provide more stability when opening the door with an elbow, as it would provide a bumper in that situation as well. Overall, the feedback we received was very helpful in furthering our design and our understanding of our problem at hand.

From this feedback, we gleaned several main guiding principles. From our understanding, our users want a handle that can be easily operated, even by one’s elbows. Another is that we need to make a design that stays on the door securely, while simultaneously being non-permanent. Additionally, one of our principles going forward is to make a universal design that works on (nearly) any door knob and only uses 3-D printed parts and common hardware. Lastly, we want to make sure to utilize a design that is easy for our team to iterate on throughout the design process.

Old “How might we” statements:
1. How might we adapt a design for multiple types of doorknobs?
2. How might we combine the idea of a foot operated door with an adapted door knob?
3. How might we weigh accessibility of materials with durability and performance?
4. How might we work with DCL to ensure our solution complies with their rules and safety regulations of the state?
New “How might we” statements:
1. How might we modify the design to better stay in place on multiple types of door knobs?
2. How might we collaborate with the maker spaces on campus to ensure anyone on campus could easily use our open-source design to make their dorm more accessible?
3. How might we work with DCL to create a more permanent option for dorms on campus?
4. How might we improve the durability of our design to withstand prolonged use?

We made some changes to our How Might We statements while collecting and analyzing feedback on our initial prototype. We found that our initial design already fit several different door knob shapes, but a more pressing issue was that it could be easily pulled or pushed out of place, or off of the door knob completely. We also decided that the durability of metal screws outweighed the problem of accessibility of materials, since most students at Brandeis who would possibly be 3D printing this design would already be in a maker space to do so. In this case, we would want to ensure that those spaces were stocked with the additional materials needed to assemble our door knob adapter.

Additionally, we still want to work with DCL to ensure that our design is safe and compliant with any rules and regulations. If possible, we want to make it an option to semi-permanently install this design in your dorm room if you want to! The last change we made was a shift towards durability if we want to be making something that could last a semester or a full year (or longer). Whether by using a stronger material or modifying our original design to be more sturdy, we want to ensure that these adapters can be used for an extended period of time if required.

This prototype is our first iteration from the pre-existing model we found online. This iteration uses the standard ~55mm diameter opening for the door knob to fit in with a modified 30mm height for the door knob to hold on to. Originally, the amount of space for the plastic to interface with the door knob was less, which resulted in lower rates of success when opening the door. As you can see in the image, the handle does not go flush to the outer ring of the circular section, this proved to be a weak point, allowing for too much flex when using the device.

As you can see in this image, we have since fixed this issue with the next version of our prototype. In addition, we have added an angle to the end of the handle to assist with operating using one's elbow, as mentioned in the analysis of the design review as well as a second screw to minimize flex and make the attachment more secured to the door. We also redrew the design in fusion in order to reduce the amount of faces and simplify the model for our own ease of prototyping going forward.
